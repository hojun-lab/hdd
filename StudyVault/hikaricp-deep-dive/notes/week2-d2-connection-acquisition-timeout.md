# Week 2, Day 2 — Connection Acquisition Flow + Timeout 구현

> 날짜: 2026-03-31
> 주제: Semaphore 기반 대기 메커니즘으로 실패율 69% → 0%

---

## 핵심 개념

### 1. 대기 메커니즘이 왜 필요한가

W2D1에서 확인한 문제:
- v1(synchronized): 6,714 req/sec, **69% 실패**
- v2(CAS only): 7,044 req/sec, **69% 실패**
- 동기화 방식(synchronized vs CAS)을 바꿔도 실패율은 동일
- **진짜 원인: 풀이 비었을 때 즉시 예외를 던지는 설계**

### 2. Polling vs Blocking

| 방식 | 설명 | 문제 |
|------|------|------|
| **Polling (Thread.sleep)** | 자고 → 확인 → 자고 → 확인 반복 | CPU 낭비, busy waiting |
| **Blocking (Semaphore)** | 허가 없으면 스레드를 재움, 반환되면 깨움 | 효율적, CPU 내놓고 대기 |

### 3. Semaphore 동작 원리

```
풀 사이즈 = 10 → Semaphore(10) → 허가증 10개

getConnection():
  tryAcquire(30s) → 허가증 1개 소모 → 커넥션 획득
  허가증 0이면 → 자동 대기 (스레드 sleep)
  30초 초과 → false 반환 → 예외

release():
  상태 NOT_IN_USE로 변경
  semaphore.release() → 허가증 1개 반환 → 대기 중인 스레드 깨움
```

### 4. acquire() vs tryAcquire(timeout)

| 메서드 | 동작 | 위험 |
|--------|------|------|
| `acquire()` | 무제한 대기 | 커넥션 안 돌아오면 **영원히 멈춤** |
| `tryAcquire(timeout, unit)` | 시간 제한 대기 | 시간 초과 시 false → 예외 처리 가능 |

프로덕션에서는 반드시 `tryAcquire` 사용 → HikariCP의 `connectionTimeout`

## 벤치마크 결과

### 테스트 조건
- 스레드 50, 풀 10, iteration 1000, 총 50,000건

### 대기 메커니즘 추가 전후 비교

| 버전 | 대기 | 처리량 | 성공 건수 | 실패 | 실패율 |
|------|------|--------|----------|------|--------|
| v2 (CAS only) | 없음 | 7,044/sec | ~15,600 | 34,392 | **69%** |
| **v2 + Semaphore(30s)** | **30초** | **3,602/sec** | **50,000** | **0** | **0%** |
| v2 + Semaphore(100ms) | 100ms | 3,195/sec | 46,413 | 3,587 | 7% |

### 핵심 발견

#### 1. 처리량이 낮아진 이유
- 이전: 실패 요청이 즉시 끝남 → 처리량 숫자 부풀려짐
- 이후: 모든 요청이 대기 → 실제 쿼리 실행 → 느리지만 전부 성공
- **실질 처리량(성공 건수 기준)은 v2+Semaphore가 3배 이상**

#### 2. connectionTimeout 트레이드오프

| 짧게 (100~1000ms) | 길게 (30000ms) |
|-------------------|----------------|
| 빠른 실패 → 빠른 에러 응답 | 느리지만 성공 |
| 장애 전파 차단 | 스레드 오래 블로킹 |
| 풀 고갈 빠른 감지 | 스레드풀까지 고갈 위험 |

- HikariCP 기본값: 30초 (정말 비정상적인 상황에서만 포기)
- 프로덕션 권장: 3~10초

## 인사이트

### 1. 대기 메커니즘 = 가장 큰 성능 개선

| 개선 | 효과 |
|------|------|
| synchronized → CAS | 처리량 5% 향상, 실패율 동일 |
| **대기 없음 → Semaphore** | **실패율 69% → 0%** |

### 2. "처리량"의 함정

처리량 숫자가 높다고 좋은 게 아니다:
- 7,044 req/sec (69% 실패) vs 3,602 req/sec (0% 실패)
- 실패한 요청은 사용자에게 500 에러
- **성공 처리량이 진짜 지표**

### 3. connectionTimeout은 SLA와 직결된다

- API SLA가 "p99 < 3초"라면 → connectionTimeout = 3초
- 3초 안에 커넥션 못 받으면 → 빠르게 실패 → 503 반환 → 클라이언트 재시도
- 30초 대기 → 스레드 30초 점유 → 톰캣 스레드풀까지 고갈 → 전체 장애
