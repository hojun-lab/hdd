> 날짜: 2026-03-25

## 핵심 개념

### 1. 커넥션 풀의 원리

```
[풀 없음 - v0]
요청 → TCP handshake → 인증 → 세션 초기화 → 쿼리 → 소켓 종료
       ~~~~~~~~~~~~~~~~~~~~~~~~ 22ms ~~~~~~~~~~~~~~~~~~~~~~~~

[풀 있음 - v1]
요청 → 풀에서 꺼내기 → 쿼리 → 풀에 반환
       ~~ 거의 0ms ~~
```

- 커넥션 생성 비용(22ms)을 **애플리케이션 시작 시 한 번만** 지불
- 이후 요청은 **이미 만들어진 커넥션을 재사용** → 생성 비용 제로

### 2. MiniPool v1 구조

```java
public class MiniPool {
    ArrayList<Connection> connectionStore;  // 보관함

    MiniPool(info, poolSize)     // 생성자: N개 커넥션 미리 생성
    synchronized getConnection() // 풀에서 꺼내기 (레이스 컨디션 방지)
    synchronized release(conn)   // 풀에 돌려놓기
}
```

- `synchronized`로 동시 접근 제어
- 비어있으면 RuntimeException (Week 2에서 대기 로직 추가 예정)

### 3. 동시성 문제: 레이스 컨디션

풀에 동기화가 없으면:
- 두 스레드가 동시에 `get(0)` → **같은 커넥션 2명이 사용**
- 하나의 소켓으로 두 쿼리가 섞임 → 프로토콜 에러 또는 결과 오염
- `synchronized`로 한 번에 하나의 스레드만 풀에 접근하도록 제한

## 벤치마크 결과: v0 vs v1

### 테스트 조건
- 스레드 50개, 각 100회, 총 5000 요청
- `SELECT 1` 실행

### 결과

| 버전 | 풀 사이즈 | 소요시간 | 처리량 | 실패 | v0 대비 |
|------|----------|----------|--------|------|---------|
| v0 (풀 없음) | - | 14.3s | 349 req/sec | 0 | - |
| **v1** | **10** | **1.05s** | **4,769 req/sec** | 0 | **13.6배** |
| v1 | 50 | 1.98s | 2,521 req/sec | 0 | 7.2배 |

### 핵심 발견

#### 1. 풀 도입만으로 13.6배 성능 향상
- v0: 매 요청마다 22ms 생성 비용
- v1: 생성 비용 0ms, 풀 접근 비용만 (synchronized 경합)

#### 2. 풀 사이즈 10 > 풀 사이즈 50 (!)
- **pool=10: 4,769 req/sec** vs **pool=50: 2,521 req/sec**
- 이유 1: 풀 사이즈 50은 초기화 시 50 × 22ms ≈ 1.1초 소모
- 이유 2: 커넥션 점유 시간이 매우 짧아서(SELECT 1 = <1ms) 10개로도 50스레드 충분
- 이유 3: 큰 리스트에서 synchronized 경합이 더 발생할 수 있음

#### 3. 소수 커넥션으로 다수 스레드 감당 가능
- pool=10, thread=50인데 실패 0
- 각 스레드의 커넥션 점유 시간이 짧으면 "돌려막기"가 됨
- **핵심 통찰: 동시 스레드 수 ≠ 필요한 커넥션 수**

## 프로덕션 시사점

| 교훈 | 설명 |
|------|------|
| 풀 사이즈는 크다고 좋지 않다 | 초기화 비용 + DB 리소스 낭비 + 경합 증가 |
| 최적 풀 사이즈는 워크로드에 따라 다르다 | 쿼리 실행 시간이 짧으면 적은 커넥션으로 충분 |
| 풀의 핵심 가치는 "생성 비용 제거" | 22ms → 0ms, 이것만으로 13배 차이 |
