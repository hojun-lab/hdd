# 커넥션 풀 직접 만들기 (1) — 349 req/sec이 4,769가 되기까지

---

## 1. 도입 — 커넥션 하나에 얼마나 걸릴까?

- "DB 연결이 비싸다"는 말, 진짜인지 직접 재봤다
- TCP 3-way handshake + MySQL 프로토콜 핸드셰이크 = 최소 4~6 왕복
- 벤치마크 결과: 평균 22ms, p99 100ms, max 358ms
- localhost인데도 22ms — 네트워크가 아니라 서버 사이드 비용이 지배적

> [시각자료] TCP + MySQL 핸드셰이크 시퀀스 다이어그램
> - Client ↔ Server 간 SYN/ACK + MySQL 인증 흐름
> - 각 단계별 소요 시간 표시
> - Mermaid sequenceDiagram 또는 draw.io 추천

> [캡처자료] 벤치마크 콘솔 출력
> - 커넥션 생성 시간 측정 결과 (avg, p50, p99, max)
> - v0 코드의 getConnection() → close() 루프 부분

## 2. 풀 없이 버텨보기 (v0)

- 구조: 매 요청마다 DriverManager.getConnection() → SELECT 1 → close()
- 50 스레드: 349 req/sec (14.3초)
- 250 스레드: 230 req/sec + 1,773 실패 — 스레드 늘렸는데 오히려 느려짐
- 매 요청마다 22ms짜리 커넥션을 새로 만드는 구조의 한계

> [시각자료] v0 아키텍처 다이어그램
> - Thread → getConnection() → TCP/MySQL Handshake → Query → close() 반복 흐름
> - 매번 새 커넥션 생성되는 걸 강조

> [시각자료] v0 벤치마크 결과 표

## 3. ArrayList 하나로 풀 만들기 (v1)

- 핵심 아이디어: N개 미리 만들어두고 빌려쓰고 반납
- 구현: ArrayList<Connection> + synchronized
- 동기화 안 하면? → 두 스레드가 같은 커넥션 훔쳐가는 레이스 컨디션
- pool=10에서 4,769 req/sec — v0 대비 13.6배

> [시각자료] v0 vs v1 구조 비교 다이어그램
> - v0: Thread --매번--> DB (새 커넥션)
> - v1: Thread --> Pool(ArrayList) --> DB (재사용)
> - 풀이 중간에서 커넥션 재활용하는 흐름

> [시각자료] 레이스 컨디션 타임라인
> - Thread A: get(0) → 커넥션 획득
> - Thread B: get(0) → 같은 커넥션 획득 ← 문제!
> - synchronized 적용 후 해결되는 흐름

> [코드블록] v1 MiniPool 핵심 코드
> - getConnection(), release() 메서드
> - synchronized 블록 위치 강조

## 4. 풀 사이즈의 역설 — 10이 50을 이긴다

- pool=10: 4,769 req/sec
- pool=50: 2,521 req/sec ← 왜?
- 원인 1: 초기화 비용 (50개 커넥션 생성 = 22ms × 50)
- 원인 2: synchronized 경합 — 풀 클수록 락 잡으려는 스레드도 많음
- 점유 시간이 짧으면(SELECT 1 < 1ms) 커넥션 10개로 50 스레드 충분

> [시각자료] 풀 사이즈별 성능 비교 막대 그래프
> | pool | req/sec |
> |------|---------|
> | 10   | 4,769   |
> | 50   | 2,521   |
> - 막대그래프로 역전 현상이 한눈에 보이게

> [시각자료] 커넥션 점유 시간 다이어그램
> - SELECT 1: 점유 < 1ms → 10개로 50스레드 커버 가능
> - 실제 쿼리: 점유 길어지면 → 10개로 부족해지는 시점

## 5. 그런데 v1은 진짜 괜찮은가?

- SELECT 1(< 1ms 점유): 200 스레드 → 0% 실패 — 괜찮아 보임
- sleep(100)(100ms 점유): 200 스레드 → **92% 실패** — 터짐
- 원인: 풀 비면 즉시 RuntimeException — 기다리는 로직이 없음
- 높은 throughput에 속지 말 것. 실패율이 진짜 지표

> [시각자료] v1 실패 시나리오 비교 표
> | 시나리오        | 스레드 | 실패율 | 원인                    |
> |----------------|--------|--------|------------------------|
> | SELECT 1       | 200    | 0%     | 점유 < 1ms, 즉시 반납  |
> | sleep(100)     | 200    | 92%    | 점유 100ms, 풀 고갈    |

> [시각자료] v1 getConnection() 흐름도
> - 풀에 남은 커넥션 있음 → 성공
> - 풀 비어있음 → 즉시 RuntimeException (대기 없음!)
> - HikariCP와의 차이점: HikariCP는 여기서 "기다린다"

> [캡처자료] sleep(100) 벤치마크 콘솔 출력
> - 실패 수(184,959)가 눈에 보이게

## 6. 다음 편 예고 — v1의 한계를 넘어서

- v1의 문제 정리:
  - 풀 비면 대기 없이 즉시 실패
  - synchronized = 글로벌 락 (모든 스레드가 한 줄로)
  - 커넥션 유효성 검증 없음
- HikariCP의 접근: ThreadLocal → CAS → Semaphore → Timeout
- 다음 글: "CAS로 바꿨는데 실패율이 똑같다고?"

> [시각자료] v1 → v2 개선 포인트 요약 표
> | v1 문제              | v2 해결 방향                   |
> |---------------------|-------------------------------|
> | 대기 없음            | Semaphore + connectionTimeout |
> | 글로벌 락            | ThreadLocal + CAS             |
> | 유효성 검증 없음      | isValid() 체크                |
